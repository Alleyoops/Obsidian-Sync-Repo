## 1）“告警查询工具（query_prometheus_alerts）”是什么意思？

**告警查询工具**本质是一个“自动化脚本/接口封装”，它去**调用 Prometheus（或 Alertmanager）的 API**，把“当前正在触发/历史触发过的告警”拉出来给人或系统看。

- Prometheus：负责**采集指标（metrics）并按规则计算是否触发告警**
    
- Alertmanager：负责**告警去重、分组、抑制、路由到通知渠道**（飞书/钉钉/Slack/短信等）
    

所以 `query_prometheus_alerts` 这种工具，通常会返回类似：

- 当前 firing 的告警列表（哪些服务、哪个实例、哪个规则触发）
    
- 告警标签（service、cluster、pod、severity…）
    
- 告警描述（annotation：summary/description/runbook）
    
- 触发时间、持续时长、触发值、阈值
    

> 面试表述：它是把监控系统里“规则算出来的异常事件”结构化拉出来，给排障流程自动用。

## 2）“日志查询工具（query_log）”是什么意思？

**日志查询工具**就是去你的日志系统（比如腾讯云 CLS、ELK/ES、Loki、Splunk 等）里检索日志并返回结果。

与告警不同：

- **告警是“指标规则”触发的事件（偏概览）**
    
- **日志是“具体请求/错误堆栈/上下文细节”（偏证据）**
    

`query_log` 做的事通常是：

- 按时间范围（最近 1 小时）
    
- 按关键词/语法（error、timeout、traceId、podName）
    
- 按字段过滤（service=xxx AND level=ERROR）
    
- 返回 Top N 条日志、聚合统计（错误次数、按错误码分布）
    

> 面试表述：告警告诉我“哪里不对”，日志告诉我“为什么不对”。

## 3）“下游发版”是什么？

**下游**指“依赖链条中你所依赖的服务/系统”（对你来说是 downstream dependency）。

举例：  
你的服务 A 调用 服务 B（下游），B 又依赖 C（更下游）。  
如果 B 或 C “发版”了（上线新版本），可能导致：

- 接口字段变更/兼容性问题
    
- 性能回退导致 A 超时
    
- 配置变更导致错误码暴增
    

所以“下游发版导致超时/错误”是非常常见的根因。

> 面试表述：排障会先看依赖链路是否近期变更（发布、配置、流量切换），因为变更是故障最大来源之一。

## 4）“context cancel / context canceled”是什么？

这是非常常见的错误，尤其在 **Go / gRPC / 微服务调用**里。

**context canceled** 通常意味着：

- 请求的上下文 `context.Context` 被取消了
    
- 原因一般是：
    
    1. **上游主动取消**（用户退出、浏览器断开、网关超时、客户端取消请求）
        
    2. **系统设置的 deadline 到了**（超时触发 cancel）
        
    3. **服务自身主动取消**（代码里调用 cancel()）
        

它本身不一定是“服务报错”，可能是：

- 上游超时了 → 取消请求 → 你这边日志出现 context canceled
    
- 你依赖的下游慢 → 导致上游超时 → 你被动 canceled
    

**面试时要点**：  
看到 context canceled，要追问“谁取消的？为什么取消？”  
通常需要结合：

- 网关/客户端超时配置
    
- 链路追踪（trace）里的耗时分布
    
- 下游依赖的延迟和错误

## 5）“用自然语言查询日志的 MCP 工具”是什么？

### MCP 是什么？

MCP（Model Context Protocol）你可以把它理解成：

> **让大模型用“统一协议”调用外部工具/数据源**的标准方式。

在你图里，“日志查询 MCP 工具”意味着：

- 大模型接收自然语言：“帮我查 ad_app 最近 1 小时 response error 的日志”
    
- MCP 工具把这句话**翻译成日志系统能执行的查询**（DSL/SQL/Lucene）
    
- 然后调用 CLS/ES/Loki 的 API
    
- 把结果结构化返回给大模型做总结/下一步决策
    

它解决的是：传统日志查询语法太复杂、门槛高；MCP 让查询“语言化、自动化”。

> 面试表述：MCP 是工具调用层，大模型负责理解意图与生成查询，工具负责访问真实系统并返回可信结果。

## 6）通篇所谓的“告警”到底是什么？

**告警（Alert）**是监控系统根据规则判断“系统状态异常”时产生的事件。

它的定位是：

- 快速发现异常（Detection）
    
- 让人/系统知道“需要行动”（Actionable）
    

告警不是日志、也不是指标本身，而是“指标/信号 + 规则”形成的“事件”。

## 7）“告警”从哪里来？怎么来的？（完整链路）

典型链路（面试高频）：

1. **应用/基础设施暴露指标**
    
    - 应用：/metrics（Prometheus 格式）
        
    - 或 SDK：OpenTelemetry/Prom client
        
2. **采集（Scrape）**
    
    - Prometheus 定时拉取指标
        
3. **规则计算（Recording/Alerting Rules）**
    
    - 例如：5xx 比例 > 1% 持续 5 分钟
        
4. **生成告警（Firing）**
    
5. **Alertmanager 处理**
    
    - 去重、合并、抑制、路由
        
6. **通知渠道**
    
    - IM 群、电话、PagerDuty、邮件
        
7. **值班响应**
    
    - runbook、自动化处置、升级
        

> 面试一句话：告警来自指标，指标来自埋点/采集；规则把连续异常转成可行动事件。

## 8）告警可能包括什么类别？（你可以用“黄金信号/RED/USE”来讲）

### A. 黄金信号（Google SRE 四大信号）

1. **Latency**：延迟（P95/P99）
    
2. **Traffic**：流量（QPS）
    
3. **Errors**：错误率（5xx、异常）
    
4. **Saturation**：饱和度（CPU/内存/线程池/队列）
    

### B. RED（服务视角）

- Rate：请求速率
    
- Error：错误
    
- Duration：耗时
    

### C. USE（资源视角）

- Utilization：利用率（CPU）
    
- Saturation：饱和（队列、负载）
    
- Errors：资源错误（磁盘 IO error）
    

> 面试技巧：用这些框架回答“你们都监控什么”，显得非常系统化。

## 9）具体会有什么告警？（给你一套“面试可直接背”的例子）

### 服务可用性

- **服务不可用 / 实例全挂**（up=0）
    
- 健康检查失败、重启频繁（CrashLoopBackOff）
    

### 错误类

- **5xx 比例升高**
    
- 特定错误码暴增（403/429/500）
    
- “依赖调用失败率”升高（db/redis/rpc error）
    

### 延迟类

- P95/P99 延迟超过阈值
    
- 超时比例升高（timeout rate）
    

### 资源类

- CPU > 80% 持续 10 分钟
    
- 内存逼近上限、频繁 GC
    
- 磁盘空间不足、inode 耗尽
    
- FD（文件句柄）耗尽
    

### 依赖类

- Redis 连接数打满/超时
    
- 数据库慢查询激增、连接池耗尽
    
- MQ 消费堆积（lag/queue depth）
    

### 业务类（更高级）

- 订单成功率下降
    
- 支付失败率上升
    
- 关键转化漏斗异常

## 10）“监控”到底监控什么？如何统计和监控的？

监控的对象大体分四层：

### 1）基础设施层

- 机器：CPU/内存/磁盘/网络
    
- 容器/K8s：Pod 重启、HPA、节点资源
    

### 2）应用层

- QPS、错误率、延迟（P95/P99）
    
- 线程池/协程、队列长度
    
- GC、内存分配
    

### 3）依赖层

- DB：连接池、慢查询、锁等待
    
- Redis：命中率、延迟、内存
    
- MQ：堆积、消费延迟
    

### 4）业务层

- 成功率、关键链路指标、SLA/SLO
    

---

### 如何统计？

核心是三种数据：

1. **Metrics（指标）**：计数器/直方图/仪表盘
    
    - counter：请求数、错误数
        
    - histogram：延迟分布（用来算 P95/P99）
        
    - gauge：CPU、内存、队列长度
        
2. **Logs（日志）**：单条事件证据
    
3. **Traces（链路追踪）**：一次请求跨服务的耗时与错误定位
    

监控系统一般是：

- 指标：Prometheus + Grafana
    
- 日志：ES/Loki/CLS
    
- Trace：Jaeger/Tempo/SkyWalking/OTel
    

> 面试一句话：我用 metrics 做趋势与告警，用 logs 找细节，用 trace 定位瓶颈与依赖问题。

## 11）结合你图里的“Executor / Replanner”，面试怎么解释更加分？

- **Executor（执行器）**：自动去查告警/查日志/查指标，把“人工操作”变成“可重复工具调用”
    
- **Replanner（动态重规划）**：根据执行结果决定下一步
    
    - 查不到日志 → 扩大时间范围/换关键词
        
    - 告警指向 CPU → 再查最耗 CPU 的进程/任务
        
    - 确认是下游发版 → 给出回滚/降级/扩容方案并终止无效步骤
        

这非常符合“自动化排障/智能运维”的思路。
