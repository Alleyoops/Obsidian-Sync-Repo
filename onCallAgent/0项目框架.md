好，我们从零开始，不先谈 prompt，也不先谈“怎么包装”，先把这个项目设计成一个**像样的系统**。

先给结论：  
这个项目不要定义成“一个会识别账号的 Agent”，而要定义成：

**一个面向规则复杂、证据不完整、误判代价较高场景的“目标账号识别决策系统”**。  
Agent 只是它的编排方式，不是项目本体。

如果一开始就把它理解成“一个 prompt + RAG + 工具调用”，后面一定越做越乱。正确做法是先把它拆成四层：**任务定义层、规则与知识层、证据处理层、决策层**，最后再在外面套一层 Agent 编排。

---

## 一、先定义项目本体：它到底在解决什么问题

这个项目不是普通检索，也不是普通分类。  
它解决的是：给定某类目标职业定义，系统需要对候选账号进行判断，输出“是否命中、依据是什么、是否需要人工复核”。

这里最关键的是把任务定义清楚：

输入不是总是完整的。有时是甲方给一批候选账号，让系统筛选；有时只给职业要求和若干描述，系统先进入候选来源模块，再对候选做识别。  
真正难的地方不在“找到文本”，而在“做判断”：

- 用户资料可能不完整
    
- profile 和动态可能互相矛盾
    
- 可能存在曾任/现任混淆
    
- 机构关联不等于个人身份
    
- 弱证据很多，强证据很少
    

所以这个系统本质上不是“生成答案”，而是**在不完全信息下做受规则约束的证据判定**。

这一定义非常重要，因为它决定了后面整个架构不会跑偏。  
一旦定义清楚，你就知道这个系统的目标不是“让 LLM 更聪明”，而是“让系统更稳、更可解释、更可复核”。

---

## 二、系统总框架：不要先想一个 Agent，要先想一条决策流水线

整个项目最合理的框架应该是：

**输入归一化 → 规则检索 → 证据抽取 → 冲突处理 → 最终判定 → 人工复核/回流**

外面再由一个 Agent 或编排器把这些模块串起来。

也就是说，真正的核心不在 Agent，而在下面这条链路：

### 第一层：输入归一化

这一层解决的是“输入不标准”的问题。

因为输入可能是：

- 一组已有账号
    
- 一个目标职业描述
    
- 一批文本化要求
    
- 多个条件混杂的任务指令
    

所以必须先做输入归一化，把所有任务转成统一格式。  
统一后的任务对象至少应该包括：

- 当前任务类型：筛选已有账号，还是处理候选列表
    
- 目标职业定义
    
- 业务上下文 / 租户信息
    
- 候选账号数据
    
- 当前需要调用的规则集版本
    

只有做完这一步，后面的模块才有稳定输入。

### 第二层：规则与知识层

这一层不是“搭个 RAG 知识库”那么简单。

规则层应该分成两部分：

一部分是**结构化规则**，比如硬条件、否决条件、时间窗口、人工复核阈值，这些适合配置化；  
另一部分是**文档化规则**，比如边界 case、冲突说明、例外情况、模糊场景解释，这些适合做成可检索规则库。

这一步的本质不是“存文档”，而是把甲方原本靠沟通和经验表达的判定逻辑，变成系统能消费的规则对象。  
如果这一步做不好，后面所有“智能识别”都只是空中楼阁。

### 第三层：证据处理层

这层是整个项目的核心，因为它决定系统是不是只会“看一眼就猜”。

证据处理层要做三件事：

第一，把候选账号的 profile、近期动态、时间信息、关联信息抽取成结构化证据。  
第二，对证据做分级，比如强证据、中证据、弱证据。  
第三，把证据中的冲突显式暴露出来，而不是让模型自己隐式猜。

这一层出来后，系统面对的就不再是原始文本，而是一组结构化证据对象。  
这非常关键，因为只有这样，你才有可能做：

- 冲突判断
    
- 置信度分层
    
- 结果复核
    
- 错误分析
    

否则你永远只是一个“大模型看原文然后给结论”的浅项目。

### 第四层：决策层

这层负责把规则和证据合在一起，输出最终判断。

它不应该完全靠模型自由发挥，而应该拆成三步：

先做**规则预判**，比如明显不满足硬条件、明显证据不足的 case 先筛出来；  
再做**规则约束下的模型决策**，让模型基于召回规则和结构化证据进行判断；  
最后做**后处理校验**，检查模型输出是否合法、是否引用了真实规则、是否存在冲突、是否应该转人工复核。

也就是说，模型只是决策层的一部分，不是全部。  
真正的系统应该是“规则、证据、模型、校验”共同完成最终判定。

### 第五层：人工复核与反馈回流

如果没有这层，这个项目就永远像 demo。

因为你的任务天然存在边界模糊和弱证据场景，不可能百分之百自动化。  
所以系统必须允许输出三种结果：

- 高置信命中
    
- 高置信不命中
    
- 待人工复核
    

同时，人工复核结果要回流到规则修订和错误分析模块。  
这一步一旦做出来，项目就从“功能型 Agent”变成“可持续优化的识别系统”。

---

## 三、Agent 在这个系统里到底扮演什么角色

现在再说 Agent，就会很清楚。

Agent 不是项目本体，也不是判断器本身。  
它更像一个**任务编排器**，负责根据当前任务情况决定系统走哪条链路。

比如：

- 如果输入已经是候选账号列表，就走“规则检索 → 证据抽取 → 判定”
    
- 如果输入信息不完整，就走“输入归一化 → 候选处理 → 证据抽取 → 判定”
    
- 如果结果低置信或冲突过强，就自动进入复核链路
    
- 如果规则召回为空，就触发兜底策略
    

也就是说，Agent 负责的是“流程控制”，不是“最终认定”。  
这样设计有两个好处：

第一，Agent 的职责清晰，不会一会儿检索、一会儿判断、一会儿复核，全混在一起。  
第二，系统可以演进。未来你想换模型、加多路召回、加版本化规则，都不会破坏整体结构。

## 四、你这个项目最应该具备的技术模块

如果把整个系统再具体一点，它至少应该有下面这些核心服务。不是让你现在全写出来，而是后面每一步补项目时，都要往这些模块上靠。

### 1. Task Service

负责接收任务请求、生成 request_id、识别任务类型、注入 tenant_id 和 rule_version。  
这是整个流程的入口。

### 2. Rule Service

负责管理规则集。内部包括：

- 结构化规则存储
    
- 文档化规则索引
    
- 规则版本管理
    
- 规则启停和租户隔离
    

它是系统的“业务逻辑底座”。

### 3. Retrieval Service

负责多路规则召回。  
不建议只做向量检索，后面最好支持：

- 向量召回
    
- 关键词召回
    
- 元数据过滤
    
- 结果融合排序
    

这会直接决定你的项目有没有“检索深度”。

### 4. Evidence Service

负责把候选资料转成结构化证据。  
输入是账号资料，输出是 evidence list。  
这一层做完，后续所有判断就不再依赖原始文本直接推理。

### 5. Decision Service

负责规则预判、模型判定、冲突处理和置信度分层。  
这是业务逻辑核心。

### 6. Review Service

负责待复核 case 的管理，包括：

- review queue
    
- 人工结果记录
    
- 回流标注
    
- 错误归因
    

### 7. Trace / Log Service

负责记录一次判定用了哪些规则、哪些证据、哪个版本、最终结果是什么。  
没有它，你后面根本没法解释系统为什么错，也没法做规则优化。

## 五、系统的数据流应该长什么样

这个项目一旦要做深，必须有清晰的数据流。你后面画架构图或者讲面试时都要围绕这个流转。

一条完整链路应该是这样：

任务进入系统后，先被归一化成统一任务对象；  
然后根据任务定义和租户信息去 Rule Service 获取结构化规则，同时由 Retrieval Service 从规则知识库里召回相关文档规则；  
接着把候选账号资料送入 Evidence Service，抽成结构化证据；  
再把“规则 + 证据”送入 Decision Service 进行决策；  
如果决策结果低置信、存在证据冲突或规则不足，就转入 Review Service；  
最后所有过程都落到 Trace/Log Service，供后续复盘、回流和版本追踪。

这个数据流的重点不是复杂，而是**每一步输入输出都独立、可替换、可解释**。  
只要你做到这点，系统自然就从“一个 prompt 工程”变成“一个可演进的业务系统”。

---

## 六、从项目建设顺序上，应该怎么落地

不要一口气做完所有高级功能。  
正确顺序应该是：

第一阶段，先做主链路可用：  
输入归一化、规则文档入库、基础检索、证据抽取、模型判定、结果输出。

第二阶段，开始做“像系统”的东西：  
结构化规则、证据分级、冲突处理、人工复核门槛、结果日志。

第三阶段，再补真正的深度：  
多路召回、规则版本化、多租户隔离、错误分析、反馈回流、评估集。

这个顺序很重要。  
因为很多人一上来就想做“多租户、多路召回、评估平台”，最后主链路都没跑通。  
你现在最需要的是先把框架搭稳，再一层一层加难点。

## 七、如果今天就把这个项目重新命名和重新定义，它应该长这样

项目名称不应该再只是“账号识别 Agent”。  
更像样的定义应该是：

**基于规则检索与证据决策的目标账号识别系统**

或者：

**面向复杂边界场景的目标账号识别 Agent 决策系统**

这两个名字比“账号识别智能体”更强，因为它们直接点出了两件事：

- 不是单纯调用模型，而是规则检索 + 证据决策
    
- 不是玩具型 Agent，而是系统

## 八、现在这一步你真正该做什么

先不要写 prompt，也不要改简历。  
先把这套框架落成一页设计稿，至少明确下面这些对象：

- 任务对象长什么样
    
- 规则对象长什么样
    
- 证据对象长什么样
    
- 判定结果长什么样
    
- 哪些服务负责哪些职责
    
- 数据在各模块之间怎么流动
    

只要这一步做好，后面你想加“多租户、多路召回、幻觉约束、review gate”，都会很顺。

下一步最合理的是，我直接带你做这个系统里的第一件实事：**先定义四个核心对象的数据结构——Task、Rule、Evidence、DecisionResult。**
