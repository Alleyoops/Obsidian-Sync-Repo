下面用 **Milvus（2.x）** 把“更新文档时如何避免查询空窗/半成品”讲到能直接落地的程度。核心就是：**不要先删旧，再建新**；要做到 **“先建新 → 原子切换 → 再回收旧”**。

---

## 方案 A（最推荐）：两套 Collection + Alias 原子切换（Blue/Green）

Milvus 有 **Collection Alias**（集合别名）。你可以把线上查询永远指向一个固定 alias，比如 `kb_chunks_active`，更新时在新 collection 构建，完成后把 alias 一次切过去。

### 数据结构

每个 chunk 至少带这些字段：

- `doc_id`（string）
    
- `chunk_id`（string）
    
- `text`（string，可放 Milvus 或放外部再存引用）
    
- `vector`（float vector）
    
- `updated_at`（int64 / datetime）
    
- （可选）`source_uri`、`page`、`section` 等
    

> 注意：Milvus 的标量过滤（expr）最好有 `doc_id` 这种字段，方便只查某份文档或某个集合子集。

### 更新流程（无空窗）

1. **当前线上：**
    

- 查询只打到 alias：`kb_chunks_active`
    
- alias 指向 `kb_chunks_v1`
    

2. **更新开始：构建新集合**
    

- 创建新集合：`kb_chunks_v2`
    
- 把新文档分块 → 向量化 → `insert` 到 `kb_chunks_v2`
    
- `flush` 确保落盘
    
- `create_index`（如果还没建）
    
- `load` 把新集合加载进内存（保证切流后不抖）
    

3. **原子切换：改 alias 指向**
    

- `alter_alias("kb_chunks_active", "kb_chunks_v2")`  
    这一步对查询来说是“瞬时切换”：要么还在 v1，要么到 v2，不会出现“查不到”的空窗。
    

4. **回收旧集合**
    

- 等一段时间（比如 10–30 分钟，或你系统的最大查询超时 + 缓冲）
    
- `release_collection("kb_chunks_v1")`
    
- `drop_collection("kb_chunks_v1")`（或保留做回滚）
    

### 查询流程（固定不变）

- 永远 query/search `kb_chunks_active` 这个 alias
    
- 可选：加过滤 `expr="doc_id == 'xxx'"`（如果一个集合装多文档）
