RAG：
分片-索引（embedding）-召回-重排-回答
这个rag有点简单了，没讲到query优化，分片策略，索引、检索结构优化，还有楼上讲的适配多种类型的文档（目前是.md）

回答：
下面是面试时可以拿得出手的一套“RAG 优化体系”讲解框架，结合你项目现状（目前只处理 `md/txt`，标题用正则切分）补足要点，避免踩坑。你可以按下面顺序答，逻辑完整、技术点密度高。

**1. Query 优化（检索前优化）**  
目标：让“检索意图更清晰、更可检索”，提升召回与精度。

- Query 重写  
把口语问题改成更“可检索”的关键词+实体+约束，比如添加领域术语、同义词、时间范围。
- Query 扩展  
自动补充同义词、缩写全称、别名、相关概念，提高召回。
- 多跳拆分  
复杂问题拆成多个子问题分别检索（例如“原因+影响+对策”）。
- 结构化增强  
如果问题明显带筛选条件（时间、部门、系统、文档类型），转成结构化过滤条件 + 向量检索。

面试点：  
你可以说“先做 query normalization / rewrite，再做检索”，这是大厂常见做法。

**2. 分片策略（Chunking）**  
目标：确保“语义完整 + 可检索 + 不丢上下文”。

- 基于结构的分片  
Markdown/HTML/PDF 按标题层级切分（项目已做了标题切分，但只用正则，层级不够严谨）。
- 段落 + 语义边界  
不仅按空行，还要避免切断列表、代码块、表格。
- 动态分片  
根据段落长度自适应，比如短段合并，长段拆分。
- Overlap 策略  
合理 overlap，保证上下文连续，但避免过多重复带来的噪声。
- 多粒度索引  
同一文档保留多个粒度 chunk（标题级 + 段落级），提高不同问题的召回概率。

面试点：  
“只按固定长度切分容易切断语义”，要结合标题/段落/结构语义。

**3. 索引结构优化**  
目标：高效检索 + 精准过滤 + 可扩展。

- 元数据设计  
包含 `_source`、`title`、`section`、`doc_type`、`created_at`、`tags` 等，支持过滤。
- 混合检索  
向量检索 + 关键词检索（BM25）结合，比单向量更稳定。
- 索引刷新策略  
全量重建 vs 增量更新，避免重复索引与脏数据。
- 冗余去重  
相同段落重复索引会污染召回，需做 hash 去重。

面试点：  
“向量检索容易语义漂移，必须配合关键词/过滤。”

**4. 检索优化（召回 + 排序）**  
目标：控制召回数量、排序更准、减少无关内容。

- 多路召回  
向量召回 + 关键词召回 + 结构化过滤召回，然后合并去重。
- Rerank  
召回后用小模型重排（cross-encoder），提升最终 top-k 质量。
- 动态 top-k  
短问题拿小 k，复杂问题加大 k，再重排。
- 评分融合  
向量相似度 + BM25 score + 文档权重合成分数。

面试点：  
“Rerank 是提升质量的关键环节”。

**5. 不同类型文档处理**  
你项目目前仅支持 `md/txt`，面试可以说“需要扩展解析链路”。

- PDF  
抽取文本 + 章节结构 + 表格，必要时 OCR。
- Word/Office  
保留标题层级、段落、列表、表格。
- HTML  
去噪、保留标题结构、过滤导航/页脚等噪音。
- 图片/扫描件  
OCR + 版面结构推断。

面试点：  
“文档类型不同，分片策略要变，不能统一按字符数切。”

**6. 评估与调优方法**  
面试官经常问“怎么评估效果”。

- 定义评价集  
问题-答案-证据对 (QA + Grounding)。
- 关键指标  
召回率、精确率、覆盖率、Rerank NDCG、最终答案一致性。
- A/B 测试  
不同分片策略、不同 top-k、不同 rerank。

面试点：  
“RAG 优化必须有评估闭环”。

- 目前只支持 `txt/md`，应扩展文档解析链路  
- 标题分片是简单正则，缺少层级树解析  
- 仅向量检索，没有 BM25/关键词检索  
- 缺乏 Rerank  
- 缺乏 query rewrite
